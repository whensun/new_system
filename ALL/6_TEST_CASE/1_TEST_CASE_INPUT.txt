INSTRUCTION: FCLASS.D
FORMAT: fclass.d rd, fs1
RULE 1: FCLASS.D classifies a double‑precision floating‑point value using the same classification categories as the single‑precision floating‑point classify instruction.
RULE 2: FCLASS.D operates on double‑precision floating‑point operands.

INSTRUCTION: FCVT.D.H
FORMAT: fcvt.d.h fd, fs1, rm
RULE 1: If the D extension is implemented, the FCVT.D.H instruction converts a half‑precision floating‑point value to a double‑precision floating‑point value.
RULE 2: The Zfhmin extension includes the FCVT.D.H instruction when the D extension is present.
RULE 3: The Zhinxmin extension includes the FCVT.D.H instruction when the Zdinx extension is present.

INSTRUCTION: VAADD.VV
FORMAT: vaadd.vv vd, vs2, vs1, vm
RULE 1: VAADD.VV computes for each active element i the sum of the signed integers in vs2[i] and vs1[i].
RULE 2: VAADD.VV performs an arithmetic right‑shift of the infinite‑precision sum by one bit for each active element i.
RULE 3: VAADD.VV rounds the shifted result for each active element i according to the current setting of vxrm.
RULE 4: The architectural effect of VAADD.VV for element i is roundoff_signed(vs2[i] + vs1[i], 1).
RULE 5: VAADD.VV performs the right‑shift and rounding on the infinite‑precision result before any truncation.
RULE 6: VAADD.VV never raises an overflow condition because the result is always representable after the rounding step.

INSTRUCTION: VADC.VIM
FORMAT: vadc.vim vd, vs2, simm5, v0
RULE 1: VADC.VIM computes for each element i of the active vector length the result vd[i] = vs2[i] + imm + v0.mask[i].
RULE 2: VADC.VIM must be encoded with the mask bit vm set to 0.
RULE 3: An encoding of VADC.VIM with vm set to 1 is reserved.
RULE 4: VADC.VIM must take its carry‑in from the mask field of the implicit vector register v0.
RULE 5: VADC.VIM does not produce a separate carry‑out; carry‑out is generated only by the companion instruction vmadc.vim.
RULE 6: VADC.VIM must not use v0 as the destination register; encodings with vd = v0 are reserved.
RULE 7: VADC.VIM must respect the source/destination overlap restrictions defined for vector instructions.
RULE 8: Although VADC.VIM is a masked instruction (vm = 0), it operates on and writes back all body elements of the destination vector register group.
RULE 9: The specification does not define a tail‑agnostic or tail‑undisturbed policy for VADC.VIM, so only body elements are written.
RULE 10: VADC.VIM uses an immediate operand that is encoded according to the vector‑immediate integer instruction format.

INSTRUCTION: VADD.VI
FORMAT: vadd.vi vd, vs2, simm5, vm
RULE 1: VADD.VI is an integer addition operation in the RISC‑V vector ISA.
RULE 2: VADD.VI uses the vector‑immediate format with operands vd, vs2, imm, and an optional mask vm.

INSTRUCTION: VAND.VI
FORMAT: vand.vi vd, vs2, simm5, vm
RULE 1: VAND.VI is the vector‑immediate form of the bitwise logical AND operation.
RULE 2: VAND.VI is listed under the Bitwise logical operations category.
RULE 3: VAND.VI can be used to invert an immediate value and then perform a bitwise AND with the source vector register vs2.
RULE 4: Because VAND.VI can invert an immediate value and AND it with vs2, a separate .vi version of the vandn instruction is not provided.
RULE 5: The functionality of a hypothetical vandn.vi instruction can be achieved by using an inversion of the immediate value with VAND.VI.

INSTRUCTION: VSUB.VV
FORMAT: vsub.vv vd, vs2, vs1, vm
RULE 1: VSUB.VV performs element‑wise integer subtraction of the corresponding elements of source vector register groups vs2 and vs1, producing a result element for each position.
RULE 2: VSUB.VV stores each result element into the corresponding element of the destination vector register group vd.
RULE 3: VSUB.VV may use an optional vector mask vm to control which result elements are written to vd; elements with a disabled mask bit are not written.
RULE 4: VSUB.VV is classified as a vector‑vector integer arithmetic operation.
RULE 5: VSUB.VV uses the assembly syntax "vsub.vv vd, vs2, vs1, vm".

RULE 10: If a floating‑point operation uses a static rounding mode encoded as 101, 110, or any other value reserved for static rounding, the behavior of that operation is reserved.
RULE 11: If the dynamic rounding‑mode field frm in fcsr contains the reserved pattern 101111, then the behavior of any floating‑point operation that depends on rounding mode is reserved.
RULE 12: If a floating‑point operation uses the dynamic rounding mode held in the frm field of fcsr, the operation uses the value of bits 7‑5 of fcsr as its rounding mode.
RULE 13: Accesses to fcsr are not visible to other harts; therefore changes made to fcsr by one hart cannot be observed by another hart.
RULE 14: Accesses to fcsr may be freely reordered with respect to FENCE instructions without violating the specification.
RULE 15: The dependency‑tracking hardware treats bits 7‑5, 4, 3, 2, 1, and 0 of fcsr as independent units.
RULE 16: The alias fflags refers to bits 4‑0 of fcsr, and the alias frm refers to bits 7‑5 of fcsr.
RULE 17: If misa.F = 1, then the FCSR bit in mstateen0 is read‑only zero, and the same bit is read‑only zero in hstateen0 and sstateen0.
RULE 18: If misa.F = 0 and a stateen0 CSR with the FCSR bit set to zero is implemented, then any floating‑point instruction causes an illegal‑instruction (or virtual‑instruction) exception as though it accessed fcsr.
RULE 19: The FCSR bit controls whether floating‑point instructions that operate on integer registers (as defined by the Zfinx and related extensions) are permitted to access fcsr.
RULE 20: If a floating‑point instruction accesses fcsr while the controlling FCSR bit is zero, the instruction causes an illegal‑instruction (or virtual‑instruction) exception.
RULE 21: The fcsr register selects the dynamic rounding mode for floating‑point arithmetic operations.
RULE 22: The fcsr register holds the accrued exception flags for floating‑point arithmetic instructions.
RULE 23: The fcsr register contains the operating mode and exception status of the floating‑point unit.

REGISTER: fflags
NUMBER: 0x001
MODE: U-mode
WIDTH: 32
FIELDS: 4..4=NV, 3..3=DZ, 2..2=OF, 1..1=UF, 0..0=NX
EXTENSIONS: D, F, Q, V
RULE 1: If a floating‑point arithmetic instruction raises an exception, the associated exception flag bit is set in fflags and remains set until cleared by software.
RULE 2: Setting a floating‑point exception flag in fflags never causes a trap to be taken.

RULE 11: If sstatus.MXR is 1, a load succeeds from pages marked readable or executable.
RULE 12: If page‑based virtual memory is not in effect, the value of sstatus.MXR has no effect on load behavior.
RULE 13: If sstatus.UBE is 0, explicit load and store memory accesses made from U‑mode are little‑endian.
RULE 14: If sstatus.UBE is 1, explicit load and store memory accesses made from U‑mode are big‑endian.
RULE 15: The value of sstatus.UBE has no effect on instruction fetches, which are always little‑endian.
RULE 16: For implicit accesses to supervisor‑level memory‑management data structures, the endianness of S‑mode applies and sstatus.UBE is ignored.
RULE 17: If an explicit CSR write sets sstatus.SDT to 1, sstatus.SIE is cleared to 0 regardless of any value written to sstatus.SIE by the same write.
RULE 18: sstatus.SIE can be set to 1 by an explicit CSR write only if sstatus.SDT is being set to 0 by the same write or is already 0.
RULE 19: When a trap is taken into supervisor mode and sstatus.SDT is currently 0, sstatus.SDT is set to 1.
RULE 20: If a trap is taken into supervisor mode while sstatus.SDT is already 1, a double‑trap exception is delivered into M‑mode.
RULE 21: When an SRET instruction is executed, sstatus.SDT is set to 0.
RULE 22: When a trap to S‑mode occurs, the sstatus.SPELP bit preserves the previous Expected Landing Pad state.
RULE 23: When a CMO instruction is executed, its permission is determined in part by the values of sstatus.SUM and sstatus.MXR (together with other relevant bits).
RULE 24: An implementation may make sstatus.UBE a read‑only field that always equals the S‑mode endianness.

REGISTER: stvec
NUMBER: 0x105
MODE: S-mode
WIDTH: 64
FIELDS: 63..2=BASE, 1..0=MODE
EXTENSIONS: H, Ssdbltrp
RULE 1: If software writes a value to stvec, the BASE field stores bits XLEN‑1 down to 2 of the supplied address and discards bits 1:0.
RULE 2: If the address stored in the BASE field of stvec is used as a trap target address, the hardware fills the lower two bits with zero to produce an XLEN‑bit address aligned on a four‑byte boundary.
RULE 3: Software MUST ensure that the address written to the BASE field of stvec is at least four‑byte aligned.
RULE 4: When the MODE field of stvec is set to Direct (value 0), all traps into supervisor mode set the program counter to the address formed from the BASE field.
RULE 5: When the MODE field of stvec is set to Vectored (value 1), all synchronous exceptions into supervisor mode set the program counter to the address formed from the BASE field.
RULE 6: When the MODE field of stvec is set to Vectored (value 1) and an asynchronous interrupt with cause number C occurs, the program counter is set to BASE plus four times C.
RULE 7: Implementations MAY require the BASE field of stvec to satisfy alignment constraints stricter than four‑byte when MODE is Vectored.
RULE 8: When M‑mode executes an MRET to return to supervisor mode, execution resumes at the address contained in stvec.
RULE 9: On delivery of a trap, the hardware does NOT apply pointer masking to the address taken from stvec.
RULE 10: If the MODE field of stvec holds a value greater than or equal to 2, the effect of traps on the program counter is architecturally reserved.

REGISTER: vcsr
NUMBER: 0x00f
MODE: U-mode
WIDTH: MXLEN
FIELDS: 2..1=VXRM, 0..0=VXSAT
EXTENSIONS: V
RULE 1: If an instruction attempts to read the CSR at address 0x00F, the read succeeds only in privilege levels that permit User‑mode CSR access; otherwise the read causes an illegal‑instruction exception.
RULE 2: If an instruction attempts to write the CSR at address 0x00F, the write succeeds only in privilege levels that permit User‑mode CSR access; otherwise the write causes an illegal‑instruction exception.
RULE 3: If the vector fixed‑point rounding‑mode field (vxrm) is changed, bits 2:1 of vcsr are automatically updated to contain the same two‑bit value.
RULE 4: If software writes a value to bits 2:1 of vcsr, the vector fixed‑point rounding‑mode field (vxrm) is set to that two‑bit value.
RULE 5: If the vector fixed‑point saturation flag (vxsat) changes state, bit 0 of vcsr is automatically updated to reflect the same state.
RULE 6: If software writes to bit 0 of vcsr, the vector fixed‑point saturation flag (vxsat) is set or cleared to match the written bit.

REGISTER: vl
NUMBER: 0xc20
MODE: U-mode
WIDTH: MXLEN
FIELDS: 63..0=VALUE
EXTENSIONS: V
RULE 1: vl may be updated only by vset{i}vl{i} configuration‑setting instructions and by fault‑only‑first vector‑load instruction variants; no other instruction may change vl.
RULE 2: After being set, vl contains an unsigned integer that specifies the number of elements to be updated by subsequent vector instructions.
RULE 3: The implementation‑dependent width of vl must be sufficient to represent all values from 0 up to the implementation’s VLMAX for the smallest supported element type.
RULE 4: An element whose index x satisfies vstart ≤ x < vl is a body element that may be processed by a vector instruction.
RULE 5: An element whose index x satisfies x ≥ vl is a tail element.
RULE 6: If vstart ≥ vl, then no body elements exist for the instruction.
RULE 7: If vstart ≥ vl, then no elements are updated in any destination vector register group, including tail elements.
RULE 8: If vl = 0, then no elements, including agnostic elements, are updated in the destination vector register group regardless of the value of vstart.
RULE 9: Tail elements (indices x ≥ vl) never raise exceptions.
RULE 10: Tail elements never update any destination vector register group unless vtype.vta = 1.
RULE 11: When vtype.vta = 1, tail elements may be overwritten with 1s or with the instruction result for mask‑producing instructions.
RULE 12: When a vsetvli, vsetivli, or vsetvl instruction executes, it writes the newly computed vl value to the destination integer register rd.
RULE 13: A vset{i}vl{i} instruction first computes VLMAX from its vtype argument before determining the new vl value.
RULE 14: For a vset{i}vl{i} instruction with rs1 ≠ x0, the AVL is taken from the source register rs1 and the resulting vl is written to rd.
RULE 15: For a vset{i}vl{i} instruction with rs1 = x0 and rd ≠ x0, the AVL is taken as the maximal unsigned integer (~0) and VLMAX is written to both vl and rd.
RULE 16: For a vset{i}vl{i} instruction with rs1 = x0 and rd = x0, the current vl is used as the AVL and the resulting vl may remain unchanged provided the new vtype does not change VLMAX.
RULE 17: If AVL ≤ VLMAX, then the resulting vl is set equal to AVL.
RULE 18: If AVL < 2 × VLMAX, then the resulting vl is chosen such that ceil(AVL/2) ≤ vl ≤ VLMAX.
RULE 19: If AVL ≥ 2 × VLMAX, then the resulting vl is set equal to VLMAX.
RULE 20: For a given implementation, the resulting vl is deterministic for identical input AVL and VLMAX values.
RULE 21: If AVL = 0, then vl is set to 0.
RULE 22: If AVL > 0, then vl is set to a value greater than 0.
RULE 23: vl is never set to a value greater than VLMAX.
RULE 24: vl is never set to a value greater than AVL.
RULE 25: If a value read from vl is later used as the AVL argument to a vset{i}vl{i} instruction and the resultant VLMAX equals the VLMAX at the time of the original read, then the new vl will be identical to the original value.
RULE 26: When an unsupported vtype value is written, the implementation sets vl to zero.
RULE 27: If a fault‑only‑first vector‑load instruction raises an exception on an element with index > 0, then vl is reduced to that element’s index.
RULE 28: If a fault‑only‑first vector‑load instruction raises an exception on element 0, then vl is not modified and a trap is taken.
RULE 29: When a fault‑only‑first instruction traps due to an interrupt, the implementation should leave vl unchanged and should set vstart instead.
RULE 30: When a debug data‑watchpoint would trigger on a tail element after the first, the implementation should leave vl unchanged and raise the debug trap.
RULE 31: For unit‑stride segment load/store instructions, vl specifies the number of segments to transfer, which equals the number of elements moved to each vector register group.
RULE 32: For whole‑register vector load/store instructions, element writes are suppressed when vstart ≥ evl (the effective vector length), regardless of the current vl value.
RULE 33: All vector instructions that reference vl treat its value as a control dependency rather than a data dependency.
RULE 34: Software may read vl with a CSR read instruction and supply that value as the AVL operand to a subsequent vset{i}vl{i} instruction, and the resulting vl will obey the same setting constraints.

REGISTER: vlenb
NUMBER: 0xc22
MODE: U-mode
WIDTH: MXLEN
FIELDS: 63..0=VALUE
EXTENSIONS: V
RULE 1: If software executes a read of CSR address 0xC22 at any privilege level, the read returns the current value of vlenb.
RULE 2: If software attempts to write CSR address 0xC22, the implementation MUST raise an illegal‑instruction exception because vlenb is read‑only.
RULE 3: The architectural value of vlenb is always equal to the implementation‑defined constant VLEN divided by 8, expressed in bytes.
RULE 4: The value of vlenb never changes during program execution; it remains constant for the lifetime of the hart configuration.
RULE 5: When software reads vlenb, the returned value can be used to compute the number of bytes transferred by whole‑register vector load or store instructions.
RULE 6: The vector‑extension status field VS includes the current value of vlenb when the VS field is saved or examined.

REGISTER: vstart
NUMBER: 0x008
MODE: U-mode
WIDTH: MXLEN
FIELDS: 63..0=VALUE
EXTENSIONS: V
RULE 1: Unprivileged software may write the vstart CSR.
RULE 2: When a trap occurs on a vector instruction, the hardware writes the element index at which the trap was taken into vstart.
RULE 3: After any vector instruction completes normally, the vstart CSR is reset to zero.
RULE 4: When a vector instruction begins execution, it starts processing elements at the index specified in vstart.
RULE 5: Vector instructions do not modify elements whose indices are less than the initial vstart value.
RULE 6: If a vector instruction raises an illegal‑instruction exception, vstart is not modified.
RULE 7: The vstart CSR provides only enough writable bits to represent indices from 0 up to VLMAX‑1.
RULE 8: Implementations SHOULD trap when software writes vstart with a value greater than the largest element index permitted by the current vtype setting.
RULE 9: Implementations are not required to trap for out‑of‑bounds vstart values.
RULE 10: Executing a vector instruction with a non‑zero vstart may cause the instruction to run slower on some implementations.
RULE 11: Certain vector instructions raise an illegal‑instruction exception if they are executed with a non‑zero vstart.
RULE 12: Implementations may raise an illegal‑instruction exception when a vector instruction is executed with a vstart value that the implementation can never produce for the same vtype setting.
RULE 13: Elements whose index is less than the initial vstart are defined as pre‑start elements.
RULE 14: Elements whose index is greater than or equal to the initial vstart and less than the current vl are defined as body elements.
RULE 15: If vstart is greater than or equal to vl, no destination vector register elements are updated by the instruction, including tail elements.
RULE 16: If vl is zero, no destination vector register elements are updated by the instruction, regardless of the value of vstart.
RULE 17: Integer and floating‑point scalar registers are written by vector instructions even when vstart ≥ vl or vl = 0.
RULE 18: On a trap caused by a vector instruction, the epc CSR is written with the address of the trapping instruction and vstart contains the element index of the trap.
RULE 19: Implementations must ensure that forward progress can eventually be guaranteed for the element index reported in vstart after a trap.
RULE 20: When a thread is migrated to a hart whose microarchitecture does not support the current vstart value, software may need to emulate execution up to a supported element position.

REGISTER: vtype
NUMBER: 0xc21
MODE: U-mode
WIDTH: MXLEN
FIELDS: 63..63=VILL, 7..7=VMA, 6..6=VTA, 5..3=VSEW, 2..0=VLMUL
EXTENSIONS: V, Zvkt
RULE 1: Only vset{i}vl{i} instructions may update vtype.
RULE 2: vtype is a read‑only XLEN‑wide CSR that can be written only by vset{i}vl{i} instructions.
RULE 3: Software SHOULD write zero to bits vtype[XLEN‑2:8]; any non‑zero value in that field is reserved.
RULE 4: Implementations MUST support the four combinations of vta and vma values (0/0, 0/1, 1/0, 1/1) as defined for handling tail and inactive elements.
RULE 5: If the vta field in vtype is 0 then tail elements are treated as undisturbed by vector instructions.
RULE 6: If the vta field in vtype is 1 then tail elements are treated as agnostic by vector instructions.
RULE 7: If the vma field in vtype is 0 then inactive (masked‑off) elements are treated as undisturbed.
RULE 8: If the vma field in vtype is 1 then inactive (masked‑off) elements are treated as agnostic.
RULE 9: If a vset{i}vl{i} instruction attempts to write an unsupported value to vtype then the vill bit in vtype is set.
RULE 10: If the vill bit in vtype is set then any vector instruction that depends upon vtype raises an illegal‑instruction exception.
RULE 11: Execution of vset{i}vl{i} and whole‑register load or store instructions does not depend upon vtype and therefore does not raise an illegal‑instruction exception when vill is set.
RULE 12: If the vill bit in vtype is set then all other bits of vtype SHALL be zero.
RULE 13: If the vtype value is not supported by the implementation then the vill bit is set, the remaining bits of vtype are zero, and the vl register is set to zero.
RULE 14: Implementations MUST consider all bits of vtype when determining whether a configuration is supported.
RULE 15: An unsupported value in any location of vtype MUST result in the vill bit being set.
RULE 16: Implementations MUST support LMUL integer values of 1, 2, 4, and 8.
RULE 17: The LMUL setting is derived from the vlmul field in vtype as LMUL = 2^{vlmul[2:0]}.
RULE 18: The element width (SEW) used for vector operations is selected by the vsew field in vtype.

REGISTER: vxrm
NUMBER: 0x00a
MODE: U-mode
WIDTH: MXLEN
FIELDS: 63..0=VALUE
EXTENSIONS: V
RULE 1: If software writes a value to the vxrm CSR, the two least‑significant bits vxrm[1:0] become the written rounding‑mode field.
RULE 2: If software writes to the vxrm CSR, the upper bits vxrm[XLEN‑1:2] should be written as zeros.
RULE 3: If vxrm[1:0] is written, the bits 2:1 of the vcsr CSR become equal to the new vxrm[1:0] value.
RULE 4: If bits 2:1 of the vcsr CSR are written, the vxrm[1:0] field becomes equal to the written vcsr bits.
RULE 5: If a rounding operation uses the rounding algorithm and the current vxrm[1:0] equals 0b00 (rnu), then the rounding increment r is the value of bit v[d‑1].
RULE 6: If a rounding operation uses the rounding algorithm and the current vxrm[1:0] equals 0b01 (rne), then the rounding increment r is v[d‑1] AND (v[d‑2:0] ≠ 0 OR v[d]).
RULE 7: If a rounding operation uses the rounding algorithm and the current vxrm[1:0] equals 0b10 (rdn), then the rounding increment r is 0.
RULE 8: If a rounding operation uses the rounding algorithm and the current vxrm[1:0] equals 0b11 (rod), then the rounding increment r is (!v[d] AND v[d‑1:0] ≠ 0).
RULE 9: If a roundoff_unsigned(v,d) function is executed, the rounding increment r used is selected according to the current vxrm[1:0] as defined in the rounding‑mode table.
RULE 10: If a roundoff_signed(v,d) function is executed, the rounding increment r used is selected according to the current vxrm[1:0] as defined in the rounding‑mode table.
RULE 11: If a vector averaging add (vaadd*) or averaging subtract (vasub*) instruction executes, the result after the one‑bit right shift is rounded by adding the rounding increment r determined from the current vxrm setting.
RULE 12: If a vector scaling right logical shift (vssrl*) or scaling right arithmetic shift (vssra*) instruction executes, the bits shifted out are rounded by adding the rounding increment r determined from the current vxrm setting.
RULE 13: If a vector narrowing fixed‑point clip (vnclip*) instruction executes, rounding around the destination least‑significant bit is performed before saturation using the rounding increment r determined from the current vxrm setting.

REGISTER: vxsat
NUMBER: 0x009
MODE: U-mode
WIDTH: MXLEN
FIELDS: 63..0=VALUE
EXTENSIONS: V
RULE 1: If a fixed‑point instruction saturates, the vxsat bit (vxsat[0]) in the vxsat CSR is set to 1.
RULE 2: If a saturating add or subtract instruction would overflow the destination, the result is replaced with the closest representable value and the vxsat bit in the vxsat CSR is set to 1.
RULE 3: If a fractional multiply with rounding and saturation instruction causes saturation, the vxsat bit in the vxsat CSR is set to 1.
RULE 4: If any destination element of a narrowing fixed‑point clip instruction is saturated, the vxsat bit in the vxsat CSR is set to 1.
RULE 5: When software writes a value to the vxsat CSR, the written least‑significant bit becomes the vxsat bit and software SHOULD write zeros to bits vxsat[XLEN‑1:1].
RULE 6: The vxsat bit in the vxsat CSR is mirrored in bit 0 of the vcsr CSR, so any change to the vxsat bit updates the vcsr vxsat bit to the same value.
RULE 7: Reads of bit 0 of the vcsr CSR return the same value as the vxsat bit in the vxsat CSR.
