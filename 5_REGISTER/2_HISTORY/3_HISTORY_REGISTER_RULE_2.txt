**Machine‑status register (mstatus)**  

*General definition*  
- **mstatus** is an MXLEN‑bit read/write CSR that holds the hart’s architectural state.  The register layout is defined in Figure 7 (RV32) and Figure 8 (RV64).  The fields and their bit positions are listed in those figures; fields may be read‑only, WARL (write‑any‑legal‑value), or WPRI (reserved, must be written with zero).  

*Global interrupt‑enable bits*  
- **MIE** and **SIE** are the global interrupt‑enable bits for M‑mode and S‑mode respectively.  They are located in the low‑order bits of mstatus so that they can be atomically set or cleared with a single CSR instruction.  
- When the hart is executing in privilege mode x, interrupts are globally enabled when **xIE = 1** and globally disabled when **xIE = 0**.  Interrupts for lower‑privilege modes (w < x) are always globally disabled regardless of any **wIE** setting; interrupts for higher‑privilege modes (y > x) are always globally enabled regardless of **yIE**.  

*Privilege‑stack fields* (Section 3.1.6.1)  
- Each privilege mode x that can respond to interrupts has a two‑level stack: **xIE**, **xPIE**, and **xPP**.  
- On a trap from privilege mode y into privilege mode x, **xPIE** is set to the previous value of **xIE**, **xIE** is cleared to 0, and **xPP** is set to y.  
- On execution of an **xRET** instruction (MRET or SRET), **xIE** is restored from **xPIE**, privilege mode is changed to the value in **xPP**, **xPIE** is set to 1, and **xPP** is set to the least‑privileged supported mode (U if U‑mode is implemented, else M).  If the new mode ≠ M, **xRET** also clears **MPRV**.  

*Double‑trap‑control (Section 3.1.6.2)*  
- The **MDT** (M‑mode‑disable‑trap) bit is a WARL field introduced by the Smdbltrp extension; it is reset to 1.  
- When **MDT = 1**, an explicit write to **MIE** clears **MIE** to 0.  For RV64 the clearing occurs regardless of the value written to **MIE** by the same CSR write.  
- **MIE** may be set to 1 only by an explicit CSR write when **MDT = 0** (or, for RV64, when the same write also sets **MDT = 0**).  
- When a trap is taken into M‑mode and **MDT = 0**, the hardware sets **MDT = 1** before delivering the trap.  If **MDT** is already 1, the trap is classified as an *unexpected trap*.  An **MRET** or **SRET** instruction clears **MDT** to 0.  

*Base‑ISA control (Section 3.1.6.3)*  
- **SXL** and **UXL** are WARL fields that control the XLEN for S‑mode and U‑mode respectively.  Their encoding matches the **MXL** field of **misa** (Table 9).  
- When **MXLEN = 32**, the **SXL** and **UXL** fields do not exist and XLEN for S‑mode and U‑mode is 32.  
- When **MXLEN = 64** and a mode is not implemented, the corresponding field is read‑only 0; otherwise it is WARL and may be hard‑wired to a value that ensures **SXLEN = MXLEN** (or **UXLEN = SXLEN**).  

*Memory‑privilege fields (Section 3.1.6.4)*  
- **MPRV** (Modify PRiVilege) modifies the effective privilege level for explicit loads and stores:  

| MPRV | MPV | MPP | Effect |
|------|-----|-----|--------|
| 0    | –   | –   | Normal access; current privilege mode applies. |
| 1    | 0   | 0   | U‑level access with HS‑level translation and protection only. |
| 1    | 0   | 1   | HS‑level access with HS‑level translation and protection only. |
| 1    | –   | 3   | M‑level access with no translation. |
| 1    | 1   | 0   | VU‑level access with two‑stage translation; HS‑level **MXR** makes any executable page readable; VS‑level **MXR** makes executable pages readable only if also readable at the guest‑physical stage. |
| 1    | 1   | 1   | VS‑level access with two‑stage translation; HS‑level **MXR** makes any executable page readable; VS‑level **MXR** makes executable pages readable only if also readable at the guest‑physical stage; VS‑level **SUM** applies instead of HS‑level **SUM**. |

- **MXR** (Make eXecutable Readable) modifies load permissions: when **MXR = 0**, loads succeed only from pages with the read bit set; when **MXR = 1**, loads succeed from pages with either the read or execute bit set.  **MXR** has no effect when page‑based virtual memory is not in effect and is read‑only 0 if S‑mode is not supported.  

- **SUM** (permit Supervisor User Memory access) modifies S‑mode load/store permissions: when **SUM = 0**, S‑mode accesses to pages accessible by U‑mode fault; when **SUM = 1**, such accesses are permitted.  **SUM** has no effect when page‑based virtual memory is not in effect; it is read‑only 0 if S‑mode is not supported or if **satp.MODE** is read‑only 0.  

*Endianness‑control fields (Section 3.1.6.5)*  
- **MBE**, **SBE**, and **UBE** are WARL fields that control the endianness of non‑instruction‑fetch memory accesses made from M‑mode, S‑mode, and U‑mode respectively (assuming **MPRV = 0** for M‑mode accesses).  **MBE = 0** selects little‑endian, **MBE = 1** selects big‑endian; analogous meanings hold for **SBE** and **UBE**.  If a mode is not implemented, the corresponding field is read‑only 0.  Implementations may make **SBE** a read‑only copy of **MBE**, and **UBE** a read‑only copy of either **MBE** or **SBE**.  

*Virtualization‑support fields (Section 3.1.6.6)*  
- **TVM** (Trap Virtual Memory) is a WARL field; when **TVM = 1**, attempts to read or write **satp**, or to execute **SFENCE.VMA**, **SINVAL.VMA**, or any instruction that accesses **satp** while executing in S‑mode raise an illegal‑instruction exception.  **TVM** is read‑only 0 when S‑mode is not implemented.  **TVM** affects only HS‑mode, not VS‑mode.  

- **TW** (Timeout Wait) is a WARL field; when **TW = 0**, the **WFI** instruction may execute in lower‑privilege modes (subject to other restrictions).  When **TW = 1**, execution of **WFI** in any less‑privileged mode that does not complete within an implementation‑specific bounded time limit causes an illegal‑instruction exception.  **TW** is read‑only 0 when there are no modes less privileged than M.

RULE 6: If the new privilege mode after an xRET instruction is not M, then the xRET instruction also clears mstatus.MPRV to 0.  
RULE 7: If mstatus.MDT = 1, then any explicit CSR write that targets mstatus.MIE forces mstatus.MIE to be cleared to 0, regardless of the written value (for RV64 the clearing occurs even if the same write also attempts to set MIE).  
RULE 8: mstatus.MIE may be set to 1 only by an explicit CSR write when mstatus.MDT = 0 (for RV64 the same write must also set mstatus.MDT = 0).  
RULE 9: When a trap is taken into M‑mode and mstatus.MDT = 0, the hardware sets mstatus.MDT to 1 before delivering the trap.  
RULE 10: If a trap is taken into M‑mode while mstatus.MDT is already 1, the trap is classified as an unexpected trap.  
RULE 11: Execution of an MRET or SRET instruction clears mstatus.MDT to 0.  
RULE 12: If mstatus.MPRV = 0, then explicit loads and stores use the current privilege mode for address translation and protection checks.  
RULE 13: If mstatus.MPRV = 1 and mstatus.MPV = 0 and mstatus.MPP = 0, then explicit memory accesses are performed with U‑level privilege using HS‑level translation and protection only.  
RULE 14: If mstatus.MPRV = 1 and mstatus.MPV = 0 and mstatus.MPP = 1, then explicit memory accesses are performed with HS‑level privilege using HS‑level translation and protection only.  
RULE 15: If mstatus.MPRV = 1 and mstatus.MPP = 3, then explicit memory accesses are performed with M‑level privilege with no address translation.  
RULE 16: If mstatus.MPRV = 1, mstatus.MPV = 1 and mstatus.MPP = 0, then explicit memory accesses are performed with VU‑level privilege using two‑stage translation, and HS‑level MXR makes any executable page readable while VS‑level MXR permits readability only if the page is also readable at the guest‑physical stage.  
RULE 17: If mstatus.MPRV = 1, mstatus.MPV = 1 and mstatus.MPP = 1, then explicit memory accesses are performed with VS‑level privilege using two‑stage translation, and HS‑level MXR makes any executable page readable while VS‑level MXR permits readability only if the page is also readable at the guest‑physical stage; VS‑level SUM applies instead of HS‑level SUM.  
RULE 18: If mstatus.MXR = 0, then a load succeeds only from a page whose read permission bit is set.  
RULE 19: If mstatus.MXR = 1, then a load succeeds from a page that has either the read or the execute permission bit set.  
RULE 20: mstatus.MXR has no effect when page‑based virtual memory is not in effect, and mstatus.MXR is read‑only 0 if S‑mode is not implemented.  
RULE 21: If mstatus.SUM = 0, then a supervisor‑mode access to a page that is accessible by user mode faults.  
RULE 22: If mstatus.SUM = 1, then supervisor‑mode accesses to pages that are accessible by user mode are permitted.  
RULE 23: mstatus.SUM has no effect when page‑based virtual memory is not in effect, and mstatus.SUM is read‑only 0 if S‑mode is not implemented or if satp.MODE is read‑only 0.  
RULE 24: If mstatus.MBE = 0, then non‑instruction‑fetch memory accesses made from M‑mode use little‑endian byte ordering (assuming mstatus.MPRV = 0).  
RULE 25: If mstatus.MBE = 1, then non‑instruction‑fetch memory accesses made from M‑mode use big‑endian byte ordering (assuming mstatus.MPRV = 0).  
RULE 26: If mstatus.SBE = 0, then non‑instruction‑fetch memory accesses made from S‑mode use little‑endian ordering (assuming mstatus.MPRV = 0).  
RULE 27: If mstatus.SBE = 1, then non‑instruction‑fetch memory accesses made from S‑mode use big‑endian ordering (assuming mstatus.MPRV = 0).  
RULE 28: If mstatus.UBE = 0, then non‑instruction‑fetch memory accesses made from U‑mode use little‑endian ordering (assuming mstatus.MPRV = 0).  
RULE 29: If mstatus.UBE = 1, then non‑instruction‑fetch memory accesses made from U‑mode use big‑endian ordering (assuming mstatus.MPRV = 0).  
RULE 30: If the hypervisor extension is implemented, then writes to mstatus.MPV record the virtualization mode (V) at trap entry.  
RULE 31: If the hypervisor extension is implemented, then writes to mstatus.GVA indicate whether the trap’s mtval contains a guest virtual address.  
RULE 32: If the hypervisor extension is implemented, then the behavior of mstatus.TSR, mstatus.TVM, and mstatus.TW is limited to affecting only HS‑mode (they do not affect VS‑mode), and mstatus.TW affects all modes except M‑mode.  
RULE 33: If mstatus.TVM = 1, then any attempt in S‑mode to read or write the satp CSR or to execute SFENCE.VMA, SINVAL.VMA, or any instruction that accesses satp raises an illegal‑instruction exception; mstatus.TVM is read‑only 0 when S‑mode is not implemented.  
RULE 34: If mstatus.TW = 0, then the WFI instruction may execute in lower‑privilege modes subject to other restrictions.  
RULE 35: If mstatus.TW = 1 and the WFI instruction is executed in any privilege mode other than M‑mode and does not complete within an implementation‑specific bounded time limit, then an illegal‑instruction exception is raised.  
RULE 36: If mstatus.TW = 1 and the WRS.NTO instruction is executed in any privilege mode other than M‑mode and does not complete within an implementation‑specific bounded time limit, then an illegal‑instruction exception is raised.  
RULE 37: When executing in VS or VU mode, if hstatus.VTW = 1, mstatus.TW is cleared, and WRS.NTO does not complete within the bounded limit, then a virtual‑instruction exception is raised.  
RULE 38: If mstatus.TSR = 1, then attempts to execute the SRET instruction while executing in S‑mode raise an illegal‑instruction exception; mstatus.TSR is read‑only 0 when S‑mode is not implemented.  
RULE 39: If the F extension is implemented, then mstatus.FS shall not be read‑only 0.  
RULE 40: If neither the F extension nor S‑mode is implemented, then mstatus.FS is read‑only 0.  
RULE 41: If S‑mode is implemented but the F extension is not, then mstatus.FS may optionally be read‑only 0.  
RULE 42: Executing an instruction that changes floating‑point state (including accesses to floating‑point CSRs) changes mstatus.FS from Initial or Clean to Dirty.  
RULE 43: If mstatus.FS = Off, then any instruction that attempts to read or write floating‑point state causes an illegal‑instruction exception.  
RULE 44: If the vector extension is implemented, then mstatus.VS shall not be read‑only 0.  
RULE 45: If neither the vector extension nor S‑mode is implemented, then mstatus.VS is read‑only 0.  
RULE 46: If S‑mode is implemented but the vector extension is not, then mstatus.VS may optionally be read‑only 0.  
RULE 47: Executing an instruction that changes vector state (including accesses to vector CSRs) changes mstatus.VS from Initial or Clean to Dirty.  
RULE 48: Implementations may also change mstatus.VS from Initial or Clean to Dirty at any time even when no vector‑state change occurs.  
RULE 49: If mstatus.VS = Off, then attempts to execute any vector instruction or to access vector CSRs raise an illegal‑instruction exception.  
RULE 50: If mstatus.VS = Dirty, then mstatus.SD = 1; otherwise mstatus.SD reflects the Dirty status of mstatus.FS and mstatus.XS according to the existing specification.  
RULE 51: If the Zfinx extension is implemented, then mstatus.FS is hard‑wired to 0 and has no effect on the trapping behavior of floating‑point instructions or fcsr accesses.  
RULE 52: The permission to execute CMO (Cache‑Management Operations) instructions may be affected by the values of mstatus.MPRV, mstatus.MPV, mstatus.MPP, mstatus.SUM, and mstatus.MXR.  
RULE 53: If the hypervisor extension is implemented, then mstatus.MPV records the virtualization mode at trap entry and is restored by MRET.  
RULE 54: If the hypervisor extension is implemented, then mstatus.GVA indicates whether mtval on a trap contains a guest virtual address.
